# 校招投递自动化 Workflow 设计

日期：2026-07-21  
状态：已完成交互确认，待用户审阅书面规格

## 1. 背景与目标

用户为 2027 届应届毕业生，需要在多家公司校园招聘官网重复完成岗位筛选、针对 JD 调整简历、填写申请表、上传附件和记录投递进度。

本项目建设一个本地优先、人在回路的求职工作流：

1. 自动发现目标公司的公开校招岗位。
2. 基于经用户确认的个人事实计算匹配度并解释原因。
3. 为用户选中的岗位生成可追溯的岗位版简历和开放题草稿。
4. 在用户已登录的招聘官网中识别并填写表单、上传岗位版简历。
5. 停在最终提交前，由用户检查并亲自提交。
6. 保存岗位、材料版本和投递状态，支持中断恢复。

首批验证公司为字节跳动、Kimi（月之暗面）、美团、拼多多、百度、快手和小红书。系统不能写死为这七家公司专用；它们用于验证通用引擎和站点适配机制。

## 2. 产品原则

- 可以少填，但不能错填；可以暂停，但不能静默跳过。
- 最终提交始终由用户触发。
- AI 可以组织和表达事实，不能创造事实。
- 个人敏感信息和招聘网站登录态保留在本机。
- 每个自动动作可解释、可审计、可恢复。
- 新公司通过岗位源插件和站点适配器接入，不修改核心业务逻辑。

## 3. 范围

### 3.1 首版包含

- 从现有 Word、PDF 简历导入个人资料。
- 一次性事实校对和后续事实维护。
- 七家目标公司公开职位的定时采集、规范化、更新和去重。
- 硬条件过滤、证据化匹配评分和推荐理由。
- 用户确认候选岗位及志愿顺序。
- 云端模型辅助生成岗位版简历、差异报告和开放题草稿。
- 本地生成并保存 PDF 和 DOCX 投递材料。
- Chrome 扩展读取当前岗位页或申请页。
- 标准表单控件识别、重复经历填写、日期转换和附件上传。
- 字段级置信度、来源和填写结果展示。
- 站点专用适配器及通用引擎降级。
- 提交前检查清单。
- 用户提交后手动标记“已提交”。
- 投递记录、执行日志和中断续跑。

### 3.2 首版不包含

- 自动点击招聘官网的最终提交按钮。
- 破解验证码、绕过风控或模拟规避检测的行为。
- 保存招聘网站密码或导出浏览器 Cookie。
- 根据邮箱自动推断笔试、面试或淘汰状态。
- 自动回答亲属关系、调剂、合规声明等高风险问题。
- 承诺覆盖所有公司、所有浏览器控件或所有页面版本。
- BOSS、智联、前程无忧等聚合招聘平台的批量触达或自动投递。

## 4. 已选技术路线

采用“本地网页控制台 + Chrome 扩展 + 云端模型 API”的分层插件架构。

未选择单体通用助手，因为页面兼容逻辑会逐渐污染核心工作流；未选择全 AI 浏览器 Agent 作为主执行器，因为其速度、成本和操作确定性不足。AI 可作为未知字段的辅助分类器，但不能替代确定性的表单执行层。

## 5. 系统架构

### 5.1 本地 Workflow Core

负责：

- 个人事实库、岗位库、申请记录和材料版本的持久化。
- 岗位采集任务、简历生成任务和表单填写任务的编排。
- 云端模型请求的脱敏、结构校验和审计。
- PDF、DOCX 生成及投递包管理。
- 提供仅监听本机的 API，供本地网页和扩展调用。

控制台是唯一业务中枢。扩展不得自行维护第二份个人资料或投递状态。

### 5.2 本地网页控制台

提供四个主要界面：

- 事实库：导入、校对和维护个人信息及经历。
- 岗位收件箱：查看匹配度、证据、截止时间和来源。
- 申请工作台：查看岗位版材料、版本差异和执行进度。
- 投递看板：查看待填写、待确认、已提交和后续状态。

### 5.3 Chrome 扩展

负责：

- 读取用户当前打开的岗位详情页和申请表页面。
- 将页面结构转换为标准字段描述，不上传密码或 Cookie。
- 从本地 Core 获取经授权的字段填写计划和附件路径。
- 执行填写、添加重复经历、选择日期、上传附件。
- 填写后重新读取页面值，记录成功、失败和页面校验错误。
- 展示提交前检查面板，但不触发最终提交。

### 5.4 云端模型 API

用于：

- 将 JD 解析为职责、硬条件、加分项和关键词。
- 生成证据化匹配解释。
- 在事实约束内提出简历选择、排序和措辞建议。
- 生成开放题草稿。
- 辅助识别通用词典未覆盖的低风险字段语义。

默认只发送 JD 和去标识化经历。姓名、电话、证件、住址、账号、Cookie 和密码不得发送。若某项生成需要真实项目名或公司名，系统应在请求预览中明确展示将发送的字段。

### 5.5 插件体系

岗位源插件实现统一接口：

- `discover()`：发现岗位。
- `fetchDetails()`：获取 JD 和元数据。
- `normalize()`：转换为统一 Job 结构。
- `fingerprint()`：生成跨入口去重标识。

站点适配器实现统一接口：

- `detect()`：判断是否匹配当前站点和页面。
- `inspect()`：补充通用表单识别结果。
- `plan()`：处理站点特殊控件或步骤。
- `execute()`：仅执行允许的局部动作。
- `verify()`：读取回填结果和页面错误。

通用引擎始终先运行。站点适配器只覆盖无法稳定处理的部分，避免复制整套填写逻辑。

## 6. 核心数据模型

### 6.1 ProfileFact

个人事实的最小可引用单位，至少包含：

- `id`
- `category`
- `value`
- `normalized_value`
- `source_document`
- `source_location`
- `verification_status`
- `sensitivity_level`
- `updated_at`

只有 `verification_status = confirmed` 的事实可以用于自动生成和高置信填写。

### 6.2 Job

至少包含：公司、岗位名称、地点、类别、职责、要求、原始 URL、首次发现时间、更新时间、截止时间、原始 JD 快照和去重指纹。

### 6.3 Application

至少包含：关联 Job、当前状态、志愿顺序、匹配报告、投递包版本、目标页面、字段填写结果、提交时间和用户备注。

状态流为：

`discovered → scored → shortlisted → package_ready → form_in_progress → review_required → submitted`

异常不创建另一条主状态链，而记录为当前步骤的 `blocked_reason`。问题解决后从当前步骤恢复。

### 6.4 GeneratedClaim

简历 bullet、开放题句子等生成内容的最小追溯单位，至少包含文本、`source_fact_ids`、模型版本、提示词版本、生成时间和用户确认状态。

### 6.5 FieldMapping

记录页面字段与事实之间的映射：页面标签、控件类型、目标事实、转换规则、置信度、风险等级、填写前值、填写后值和验证结果。

## 7. 用户工作流

### 7.1 初次设置

1. 导入 Word 和/或 PDF 主简历。
2. 系统解析并生成事实候选项。
3. 用户处理冲突、补充缺失项并逐项确认。
4. 配置求职偏好、硬性排除条件、云端模型 API Key 和目标公司。
5. 系统生成一份基线简历供用户确认。

### 7.2 日常搜岗与筛选

1. 岗位源插件定时获取公开职位。
2. 系统规范化、去重并保存 JD 快照。
3. 硬条件规则先排除明显不符合的岗位。
4. 模型基于事实证据生成匹配评分和解释。
5. 用户在候选岗位收件箱中选择申请项并设置志愿顺序。

匹配分数只用于排序，不得代替用户决定，也不得以虚构关键词提高分数。

### 7.3 生成投递包

1. 选择与 JD 最相关的已确认事实。
2. 生成简历内容建议及每条内容的事实引用。
3. 校验数字、日期、实体、技能和来源。
4. 对无来源或改变关键事实的输出拒绝生成。
5. 生成差异视图，由用户查看新增表达、删除内容和顺序变化。
6. 输出 PDF、DOCX、开放题草稿和匹配报告。

投递包目录逻辑上包含：

```text
company-role/
├── jd-snapshot.txt
├── match-report.json
├── resume.pdf
├── resume.docx
├── form-answers.json
└── manifest.json
```

### 7.4 表单填写

1. 用户在浏览器中登录招聘官网并打开申请页面。
2. 扩展识别页面和适配器。
3. Core 返回字段填写计划和当前岗位附件。
4. 扩展逐字段执行并立即回读验证。
5. 页面包含多步表单时，每步完成后保存 checkpoint。
6. 所有可处理页面结束后进入 `review_required`。
7. 用户检查红、黄、绿三类字段及附件。
8. 用户亲自在官网提交，再点击扩展的“已提交”更新记录。

## 8. 字段风险与置信策略

### 8.1 高置信度

来源明确、语义唯一且格式转换确定的字段，例如姓名、学校、专业、明确起止日期。系统自动填写并回读验证。

### 8.2 中置信度

存在可解释映射或格式推断但不完全唯一的字段。系统可填写，但必须标黄并在提交前逐项确认。

### 8.3 低置信度或高风险

包括但不限于志愿调剂、亲属关系、竞业、证件声明、政治面貌的歧义选项、工作授权、合规承诺以及语义无法确定的字段。系统保持为空并要求用户处理。

风险等级优先于置信度：即使模型对高风险问题很有把握，也不得自动作答。

## 9. 失败处理与恢复

- 登录过期或验证码：暂停当前任务，用户完成后从当前页面继续。
- 页面改版：通用引擎尝试安全降级；保存脱敏 DOM 特征、失败字段和错误信息，用于修复适配器。
- 模型限流或超时：指数退避重试；仍失败时允许用户稍后重跑，不丢失上游结果。
- 文件生成失败：申请保持在生成步骤，不允许进入填写状态。
- 附件上传失败：申请不得进入 `review_required` 的可提交状态。
- 页面校验失败：记录字段级错误并聚焦到对应字段。
- 重复岗位：合并来源和更新时间，不创建重复申请。
- 网站下架岗位：保留历史快照并标记失效，不自动删除申请记录。

## 10. 安全与隐私

- 本地服务仅绑定回环地址并要求扩展握手令牌。
- API Key 使用操作系统安全存储；不得写入仓库、日志或浏览器页面。
- 招聘网站登录完全由浏览器管理。
- 日志默认脱敏，禁止记录证件号、完整电话、Cookie、密码和文件原文。
- 扩展权限按域名和用户动作最小化申请。
- 模型调用保存字段清单、模型名和时间，但不重复保存敏感请求正文。
- 删除个人资料时，应同时删除生成材料和本地索引，并给出明确确认。

## 11. 测试策略

### 11.1 自动测试

- 事实导入、冲突检测和敏感字段分类单元测试。
- JD 规范化、去重和岗位状态变化测试。
- 生成结构、事实引用、数字和日期一致性测试。
- 标准 HTML 控件、动态下拉框、日期控件、重复区块和附件上传的扩展测试。
- 岗位源插件与站点适配器的契约测试。
- 使用脱敏且获准保存的页面夹具执行回归测试。
- 中断、重试和 checkpoint 恢复测试。

### 11.2 真实站点提交前演练

对七家首批公司各选择至少一个真实公开岗位，完成：采集、匹配、生成、打开申请页、填写、上传、检查；测试到最终提交按钮前停止。

若网站需要真实账号或验证码，由用户亲自完成。测试材料使用用户确认的真实资料，任何测试投递都不得自动提交。

## 12. 首版验收标准

- 七家目标公司的公开岗位均能采集、规范化并去重；若公司主动禁止自动访问，则改为用户打开岗位页后一键采集，并记录为已知限制。
- 标准字段自动填写正确率不低于 90%。正确率按成功回读且与计划值相同的字段数计算。
- 高风险字段自动猜测次数为零。
- 岗位版简历和开放题中的事实性陈述 100% 有 `source_fact_ids`。
- 生成内容不得改变已确认的数字、日期、单位、实体和技能边界。
- 多步表单中断后可从最近 checkpoint 恢复。
- 附件上传失败、必填项未完成或页面仍有校验错误时，不显示“可提交”。
- 扩展不存在点击最终提交按钮的代码路径。

## 13. 分阶段交付

### 阶段一：可信数据与简历引擎

完成事实导入、校对、JD 解析、匹配、可追溯生成和 PDF/DOCX 输出。这是独立可用的第一个子项目。

### 阶段二：通用浏览器填表

完成扩展、本地通信、标准控件填写、附件上传、字段验证和提交前检查。

### 阶段三：搜岗插件与首批站点适配

接入七家公司公开岗位源；使用真实失败样本为复杂申请页增加小型适配器。

### 阶段四：投递管理与可靠性加固

完善看板、日志、恢复、回归夹具和兼容性监控。

各阶段都应可独立验收，不以全链路全部完成作为首次可用条件。

## 14. 后续候选能力

以下能力不进入首版承诺，待真实使用数据验证价值后再决定：

- 从邮件中提取投递进度，先建议、后由用户确认更新。
- 更多 ATS 和公司官网适配器。
- 本地 Ollama 模型选项。
- 面试日历和提醒。
- 岗位变化监控及新增岗位通知。
- 不同简历策略的回复率分析。
